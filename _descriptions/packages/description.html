The DESCRIPTION file contains basic information about your package. This information is specified by predefined fields. Some of these fields are required and others are only optional. Explore these fields by clicking on them in the example file.

The Package field specifies the name of your package. The name of a package must be at least two letters long, contain only letters, numbers, and dots, and start with a letter.

However, you are encouraged to not use dots and to only use lowercase letters. Numbers are still allowed, but use them sparingly.

* required field

The Type field specifies the type of the package. You will always specify the package type as Package and if this field is ommited the package type is assumed to be Package.

The Title field gives a short description of the package. Because this short description is often displayed alongside the package name you are advised to exclude repeating the name in the description. The title should be written in title-case, i.e. capitalize principal words, but should not end in a period.

* required field

The Version field specifies the version of the package. The default version is often 0.1.0, but you are encouraged to start at version 0.0.1. When a significant update to the package is made, be sure to bump the version number. Your first bump might be from 0.0.1 to 0.0.2.

Often, versions beyond 0.1.0 are considered stable. Those versions 0.0.9 and below could be considered alpha or beta versions. For finer control over your development versions consider starting with version 0.0.01.

When you release your package, you may consider bumping your package version to 0.1.0. However, if you are publishing your R package, but still unsure of the toolset provided consider keeping your version below 0.1.0.

* required field

The Author field specifies who is the package author, first and last name.

* required field (unless the Authors@R field is specified)

The Maintainer field specifies who is maintaining the package. As you can see in the example the field value is a full name followed by an email. Be sure the email is surrounded by angle brackets.

The maintainer serves as a point of contact for bug reports. See the BugReports field for more information.

* required field (unless the Authors@R field is specified)

The Authors@R field may be specified instead of the Author and Maintainer field.

The value of this field is R code and is one or more calls to person(). See ?person for more information about the arguments. There are many many possible values for the role argument, but the roles you will most often use are,

"cre"
indicating a package creator
"aut"
indicating a package author, someone fundamental to the development of the package
"ctb"
indicating a package contributor, someone who has made significant contributions to the package

For all possible roles see https://www.loc.gov/marc/relators/relaterm.html .

The Description field is a long-form description of the package. You may be tempted to start the description with "<package> is …" or "This package is …", but CRAN asks you avoid doing so. Additionally, when writing the description bear in mind that the description is intended for R programmers of all levels and should be a single paragraph. An in-depth, technical explanation of your package may be more suited for a vignette.

Any URLs specified must be wrapped in angle brackets (< and >). If you would like to wrap the paragraph's sentences onto new lines make sure each line is indented with 4 spaces.

* required field

The URL field specifies one or more URLs associated with your package. If multiple URLs are specified they must be separated by a space or comma. If your package code is on GitHub be sure to include the repository URL.

Be sure to include the scheme prefix, "http://" or "https://", in each of your URLs. Additionally, any spaces in a URL must be replaced with "%20".

* recommended field

The BugReports field specifies a single URL where bug reports may be submitted. Like the example here, if your project is on GitHub the bug reports URL should be the issues page of the repository.

* recommended field

The License field specifies the licensing of your package. Specifying a license is important so others know how and when they may use your package.

The R-project offers the following standard license shorthand specifications. Each shorthand is associated with a document outlining the copyright holder and the terms and conditions.

GPL-2
https://www.r-project.org/Licenses/GPL-2
GPL-3
https://www.r-project.org/Licenses/GPL-3
LGPL-2
https://www.r-project.org/Licenses/LGPL-2
LGPL-2.1
https://www.r-project.org/Licenses/LGPL-2.1
LGPL-3
https://www.r-project.org/Licenses/LGPL-3
AGPL-3
https://www.r-project.org/Licenses/AGPL-3
Artistic-2.0
https://www.r-project.org/Licenses/Artistic-2.0
BSD_2_clause
https://www.r-project.org/Licenses/BSD_2_clause * template
BSD_3_clause
https://www.r-project.org/Licenses/BSD_3_clause * template
MIT
https://www.r-project.org/Licenses/MIT * template

The last 3 licenses are templates and allow you to specify a copyright holder and year of copyright. When using a license template you must include a LICENSE file in your package's top-level directory (where the DESCRIPTION file is) and append "+ file LICENSE" to the field's value. Each template outlines what information must be included in the LICENSE file.

* required field

The Encoding field is necessary if your DESCRIPTION file uses non-ASCII characters. Put another way, if you are writing your DESCRIPTION file using only the standard characters on an English computer keyboard you do not need to include this field.

The Imports field specifies which packages are used by your package. Any package from which you use a function must be included in this field. The only package which you do not have to list is the base package.

The listed packages must be separated by commas, appear on separate lines starting below the field name, and must be indented by 4 spaces.

The Suggests field specifies 0 or more packages used in your package's examples, tests, or vignettes. Remember, a package should only be listed in one of Imports, Depends, or Suggests.

The value of this field follows the same structure as the Imports field.

The Depends field specifies 0 or more packages to attach prior to attaching your package (via library or require). If your package's use case heavily incorporates functions from another package (think, only works in conjunction with these functions), then consider including the package in this field.

A package may be listed twice in Depends to limit a package within a range of versions. When using the devtools or roxygen2 packages you may see a a version of R automically added to this field.

The value of this field follows the same structure as the Imports field.
