The DESCRIPTION file contains basic information about your package. This information is specified by a predefined set of fields. Some of these headers are required, marked with a *, and others are only optional.

The Package field specifies the name of your package. The name of a package must be at least two letters long, contain only letters, numbers, and dots, and start with a letter.

However, you are encouraged to not use dots and to only use lowercase letters. Numbers are still allowed, but use them sparingly.

* required

The Type field specifies the type of the package. You will always specify the package type as Package and if this field is ommited the package type is assumed to be Package.

The Title field gives a short description of the package. Because this short description is often displayed alongside the package name you are advised to exclude repeating the name in the description. The title should be written in title-case, i.e. capitalize principal words, but should not end in a period.

* required

The Version field specifies the version of the package. The default version is often 0.1.0, but you are encouraged to start at version 0.0.1. When a significant update to the package is made, be sure to bump the version number. Your first bump might be from 0.0.1 to 0.0.2.

Often, versions beyond 0.1.0 are considered stable. Those versions 0.0.9 and below could be considered alpha or beta versions. For finer control over your development versions consider starting with version 0.0.01.

When you release your package, you may consider bumping your package version to 0.1.0. However, if you are publishing your R package, but still unsure of the toolset provided consider keeping your version below 0.1.0.

* required

The Author field specifies who is the package author, first and last name.

* required (unless the Authors@R field is specified)

The Maintainer field specifies who is maintaining the package. As you can see in the example the field value is a full name followed by an email. Be sure the email is surrounded by angle brackets.

The maintainer serves as a point of contact for bug reports. See the BugReports field for more information.

* required (unless the Authors@R field is specified)

The Authors@R field may be specified instead of the Author and Maintainer field.

The value of this field is R code and is one or more calls to person(). See ?person for more information about the arguments. There are many many possible values for the role argument, but the roles you will most often use are,

"cre"
indicating a package creator
"aut"
indicating a package author, someone fundamental to the development of the package
"ctb"
indicating a package contributor, someone who has made significant contributions to the package

For all possible roles see https://www.loc.gov/marc/relators/relaterm.html.
